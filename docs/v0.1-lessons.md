# v0.1 dogfood — lessons from the dry-run

Notes captured during the macOS dry-run of installer v0.1.0. The plan (`docs/superpowers/plans/2026-05-16-bsky-saves-install-v0.1.md`) and design spec (`docs/superpowers/specs/2026-05-16-bsky-saves-install-v0.1-design.md`) reflected the design as understood from the bootstrap briefs; these are the things the briefs didn't tell us, the implementation got wrong, or only surfaced when actually running on a Mac. Useful as input to the public-release-milestone spec.

## Architectural surprises

### 1. Briefcase's macOS bundle ships no standalone `python3`

The bootstrap brief implied a normal Python installation inside the `.app`. In practice, the Briefcase macOS bundle exposes a Python framework (`Contents/Frameworks/Python.framework/Versions/3.11/`) **with no CLI binary** — only the framework's dylib. The only "Python interpreter" inside the bundle is the `.app`'s stub at `Contents/MacOS/<formal_name>`, which is also `sys.executable` from Python's perspective.

**Consequence:** subprocess-based supervision (the original spec § 2 design) is impossible inside the bundle. Spawning `sys.executable` re-launches the whole `.app` — we fork-bombed the launcher on first attempt. The fix was to replace the subprocess supervisor with an in-process daemon thread that calls `bsky_saves.cli.main(["serve", "--gui"])` directly.

**For the public-release spec:** the launcher↔helper isolation model in v0.1 design § 2 needs revision. The "extension path" via helper HTTP control endpoints still stands, but the day-1 architecture is single-process. If we want real isolation, options are (a) ship a custom support-binary alongside the Briefcase stub, (b) use a different freezing tool that bundles a CLI python (PyOxidizer? PyInstaller?), or (c) accept single-process as the long-term model and lean into it.

### 2. Tkinter and pystray cannot share the macOS main runloop

Plan Task 5 specified a Tkinter status window. On macOS, Tk and pystray both require macOS's main runloop and don't cooperate — once `pystray.Icon.run()` is called, calling `tk.Tk()` from a menu callback creates a window that never renders (no `mainloop()` is running; calling it would deadlock). The placeholder window the plan called for *silently did not appear*.

**Fix in v0.1:** replaced Tkinter with a native `osascript display dialog` for the placeholder. Lightweight, no runloop conflict, fine for "click OK to dismiss" UX.

**For the public-release spec:** a richer status window on macOS needs to use either (a) NSWindow via PyObjC (pystray already pulls in `pyobjc` on macOS, so the dep is "free") or (b) a webview (`pywebview`) which manages its own native window. Whichever we pick, Tk is off the table.

### 3. pystray's "default action on icon click" is Linux/Windows-only

The plan put "Show status..." on a `pystray.MenuItem(..., default=True, visible=False)` entry, expecting left-click to invoke it. On macOS this is unreachable — clicking a menu-bar icon always just opens the menu. We surfaced "Show status…" as a regular visible item.

**For the public-release spec:** assume the tray icon is *only* a menu launcher on macOS. Don't design click-target affordances that hide behind a default action.

## Build-pipeline surprises

### 4. Briefcase enforces app-key ↔ package-name parity

`[tool.briefcase.app.<key>]` requires `sources` to include a package named `<key>` with hyphens swapped for underscores. With key `bsky-saves`, Briefcase looked for a `bsky_saves` package and our `bsky_saves_launcher` package failed validation. We renamed the key to `bsky-saves-launcher` (the user-facing `Bsky Saves.app` name comes from `formal_name`, which is independent).

### 5. Briefcase requires `__main__.py`

Briefcase launches the app via `python -m <package>`. Without `<package>/__main__.py`, the `.app` crashes on launch with `'<package>' is a package and cannot be directly executed`. The plan covered `app.py` with `def main()` but didn't anticipate the `__main__.py` shim. Added it as a tiny re-export of `app.main()`.

### 6. `pip install file://./...` doesn't work; use `PIP_FIND_LINKS`

The plan pinned the wheel via `requires = ["bsky-saves @ file://./wheelhouse/..."]`. Pip rejects relative `file://` URLs with `non-local file URIs are not supported`. Absolute paths would be machine-specific and not commit-friendly.

**Fix:** pin by version (`"bsky-saves==X.Y.Z"`) and set `PIP_FIND_LINKS=./wheelhouse` on the build step. Pip prefers the local SHA-verified wheel and falls back to PyPI for transitive deps. `fetch_wheel.py` remains the SHA gate.

### 7. The bootstrap doc's `/ping` payload shape was misleading

GUI bootstrap § 2 listed `/ping` returning `{name, version, protocol, gui_bundled, features[]}` and I read `gui_bundled` as a boolean. The actual payload uses `gui_bundled` as the **version string of the bundled GUI** (e.g. `"0.6.2"`), falsey if no GUI is bundled. The v0.1 release.yml smoke assertion `assert d.get('gui_bundled') is True` could never pass; this is why the first v0.1.0 release build failed. Fixed in `release.yml` to check truthiness + log the version.

**For the public-release spec:** any helper-side documentation we write or consume should be specific about field types. Don't infer from prose.

### 8. `bsky-saves serve` without `--gui` returns auth-required at `/`

Bare `bsky-saves serve` exposes only the auth-protected API. The bundled GUI is served only when `--gui` is passed, which also handles the `__BSKY_SAVES_TOKEN__` sentinel substitution in `index.html` so the GUI is pre-paired. We had to add `--gui` to `HELPER_ARGV`.

### 9. GitHub asset filenames replace spaces with dots in URLs

Briefcase produces `Bsky Saves-0.1.0.dmg` (formal_name includes a space). GitHub displays the original name on the release page but rewrites it to `Bsky.Saves-0.1.0.dmg` in download URLs. `SHA256SUMS` lists the space-named file, so `curl -O <url> && shasum -c SHA256SUMS` fails to find it. Fixed in v0.1.1 by renaming `.dmg`s to remove spaces before hashing + uploading.

## Process surprises

### 10. AppleScript tab-focusing has a friction tax

We prototyped an `osascript`-based "focus existing GUI tab if one is open in Safari/Chrome" handler to avoid the "Open GUI" → new-tab-every-click pattern. It works, but the first invocation per browser triggers a macOS Automation permission prompt ("Bsky Saves wants to control Safari"). For a dogfood release that surprises the user; pulled the feature, kept the code in a comment block in `tray.py`.

**For the public-release spec:** the macOS Automation permission story needs more thought before we ship anything that drives other apps. The same problem will hit "Open hosted version" if we ever implement focus-existing.

### 11. The plan's run-twice-on-clean-machine assumption was correct

The dry-run on tenorune's Mac surfaced multiple issues the CI runner would have missed (or would have hit later in more confusing ways): pip-version preconditions, Python-version preconditions, Briefcase config errors, the fork-bomb, Tk/pystray runloop, etc. **The "spike one OS locally before tagging" gate in the runbook earned its keep.** Keep it for the public-release milestone too — adding Windows/Linux to CI without a local dry-run on each would have shipped broken artifacts.

## Things that worked unchanged from the plan

- `scripts/fetch_wheel.py` (TDD, SHA verification, zero-SHA sentinel) — landed as written, behaved exactly as designed.
- The cross-repo `wheel-version-bump` dispatch shape — helper-team PR #18 implemented our payload contract verbatim.
- `ci.yml` lint+test on macos-latest — passed on every PR without modification.
- `.github/workflows/release.yml` skeleton — survived all the fix iterations; only the smoke assertion and the filename-space rename needed correction.
- Briefcase + pystray as the bundler / tray-library pair — both are the right call. Replacing either at this point would be a larger undertaking with no upside for v0.1.
